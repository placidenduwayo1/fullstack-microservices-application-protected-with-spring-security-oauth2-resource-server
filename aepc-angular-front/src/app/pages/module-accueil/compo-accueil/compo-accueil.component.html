• amazing microservices applications compared to monolithic applications
  • here are the application demo with apis business microservices
    • api-address: to manage addresses
    • api-company: to manage companies
    • api-employee: to manage employees
    • api-project: to manage projects
  • in this demo, each api business microservices has its own dependencies: db, configurations, etc
  • each api business microservices of the application is implemented in clean-architecture
  • each couple of apis: (api-company,api-address) and (api-emmployee,api-address), (api-project, api-company),(api-project, api-emmployee) communicates. That is, to validate:
    • a company, api-company gets address from api-address
    • a employee, api-employee gets address from api-address
    • a project, api-project gets company and employee from company, employee apis
  • on start up each backend api microservices gets its configurations from a api-config service
    • api-address: to manage addresses
    • api-company: to manage companies
    • api-employee: to manage employees
    • api-project: to manage projects
    • api-securty-service: to authenticate users'requests on the applicaton
    • api-registry-service: to register apis with their name in a registry
    • api-gatway-service: to route request from frontend to backend
  • after they get their config, backend apis microservices register on api-registry-service
  • a proxy api is configured to route request front (angular appli) to backend
  • our fronted is an angular application
  • each request from frontend or between apis that communicate must be authenticated
    • a jwt token must be provided within http request
    • a remote server for which destined the request decode the jwt token
