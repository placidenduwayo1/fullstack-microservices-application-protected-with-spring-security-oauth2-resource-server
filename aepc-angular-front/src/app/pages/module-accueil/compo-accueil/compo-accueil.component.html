• microservices based applications are attractive compared with monolithic ones
  • here are the microservices based application demo with business microservices apis:
    • api-address: to manage addresses
    • api-company: to manage companies
    • api-employee: to manage employees
    • api-project: to manage projects
  • in this demo, each business microservice api has its own dependencies: db, configurations, etc
  • each business microservice api is implemented in clean-architecture
  • elements in each couples: (company, address), (employee, address), (project, company), (project, emmployee) communicates. That is, to validate:
    • a company, api-company gets address from api-address
    • a employee, api-employee gets address from api-address
    • a project, api-project gets company and employee from company, employee apis
  • api-config-service: on start up each backend microservice api gets its configurations from a config service:
    • business microservices: api-address, api-company , api-employee, api-project
    • security microservice: api-securty-service: to authenticate users'requests
    • microservices registration service: api-registry-service: to register apis with their name
    • gateway service api-gatway-service: stands between frontend (Angular application) and backend ( microservices based application)and route users' requests
  • api-registry-service: when they get configurations, microservices apis register in registry
  • api-gatway-service: we created and configured a gateway to route angular application requests
  • api-security-service: requests from frontend or between microservices apis are authenticated:
    • a jwt token must be provided within http request
    • a remote servirce for which is destined the request decode the jwt token
  • we created our fronted application with angular
